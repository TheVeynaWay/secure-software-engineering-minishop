## Parts Initially Not Understood

The following are parts of the AI-generated code that were not immediately obvious and required review to understand:

### 1. `@wraps(f)` inside decorator functions (`app.py` lines 15–21, 24–34)

```python
from functools import wraps

def login_required(f):
    @wraps(f)
    def decorated(*args, **kwargs):
        ...
    return decorated
```

`@wraps(f)` copies the original function's name and docstring onto the wrapper. Without it, Flask's internal routing table sees every decorated route as the same function (`decorated`), causing an `AssertionError` about duplicate endpoint names. This was not obvious until reading the Flask documentation.

### 2. `db.row_factory = sqlite3.Row` (`models.py` line 9)

Setting `row_factory` makes each returned row behave like a dictionary, so templates can access columns by name (e.g., `product['name']`) instead of by index (`product[1]`). Without this line, Jinja templates would break when trying to access named columns.

### 3. `session.clear()` before re-populating session (`app.py` line 100)

```python
session.clear()
session['user_id'] = user['id']
```

Calling `session.clear()` first is a defense against session fixation attacks, where an attacker pre-sets a known session ID before the victim logs in. This pattern was in the generated code but its security purpose was not immediately obvious.

### 4. `generate_password_hash` / `check_password_hash` (Werkzeug)

The generated code uses Werkzeug's hashing utilities rather than calling `hashlib` directly. Werkzeug automatically salts the password and encodes the algorithm name into the stored hash string (e.g., `scrypt:...`), so no separate salt column is needed. Understanding what the stored hash string contains required reading Werkzeug's documentation.

## Issues and Limitations with AI-Generated Code

The items below are intentional simplifications acceptable for a course project, but would be problems in a production system. They are documented here so the reviewing team knows what to look for.

### Security Limitations

| Issue | Location | Description |
|-------|----------|-------------|
| Hardcoded `secret_key` | `app.py` line 7 | `app.secret_key = 'change-this-in-production'` is a static string. In production this must be a long random value loaded from an environment variable, otherwise session cookies can be forged. |
| No CSRF protection | All POST forms | Flask-WTF (which adds CSRF tokens) was intentionally excluded to keep scope small. All state-changing POST routes are vulnerable to cross-site request forgery. |
| No rate limiting on login | `app.py` `/login` route | An attacker can send unlimited login attempts. No lockout, delay, or CAPTCHA is implemented. |
| Weak seeded admin password | `models.py` line 46 | The default admin password `admin123` is very weak and is visible in source code. |
| No session expiry | `app.py` | Sessions persist until the browser closes. There is no server-side timeout or `PERMANENT_SESSION_LIFETIME` configured. |
| No maximum input length enforcement | Registration / product forms | The server checks minimum lengths but does not cap maximum input length, which could allow oversized payloads. |

### Functional Limitations

| Issue | Description |
|-------|-------------|
| No pagination | Product and order lists load all rows from the database at once. |
| Deleting a product orphans orders | `admin_delete_product` does not check whether open orders reference the product. The JOIN in order history would still work because of SQLite's deferred FK enforcement, but the product name would be lost. |
| Quantity validation gap | The order form validates that quantity is a positive digit string but does not enforce an upper bound, so a user could submit an arbitrarily large quantity. |

## What Was Reviewed and Verified After Generation

- Confirmed passwords are hashed (not plaintext) by inspecting `models.py` and checking the database with a SQLite browser.
- Confirmed all protected routes use `@login_required` or `@admin_required` decorators by reading every route in `app.py`.
- Confirmed SQL queries use `?` parameterized placeholders throughout — no raw string formatting is used in any query.
- Confirmed admin-only routes return 302 redirects (not just hide buttons) when accessed by a non-admin by testing directly in the browser.
